# Debug 方法论（借鉴 gstack investigate）

## 铁律：不查清根因不修

修症状 = 打地鼠。每次修症状不修根因，下一个bug更难找。

## Phase 1: 根因调查

1. **收集症状** — 读错误信息、堆栈、复现步骤。不够就一次问一个问题
2. **读代码** — 从症状沿代码路径回溯，grep所有引用，读逻辑
3. **查最近改动** — `git log --oneline -20 -- <files>`，回归=根因在diff里
4. **复现** — 能确定触发吗？不能就继续收集证据

输出：**"根因假设：..."** — 具体、可测试的声明

## Phase 2: 模式分析

| 模式 | 特征 | 查哪里 |
|------|------|--------|
| 竞态条件 | 间歇性、时序相关 | 共享状态的并发访问 |
| 空值传播 | NoMethodError/TypeError | 缺少optional guard |
| 状态损坏 | 数据不一致、部分更新 | 事务、回调、钩子 |
| 集成失败 | 超时、意外响应 | 外部API、服务边界 |
| 配置漂移 | 本地正常/staging挂 | 环境变量、feature flag |
| 缓存过期 | 显示旧数据、清缓存后正常 | Redis/CDN/浏览器缓存 |

同一文件反复出bug = 架构臭味，不是巧合

## Phase 3: 假设验证

1. 确认假设 — 加临时日志/断言，跑复现，证据是否匹配？
2. 假设错误 → 回Phase 1，收集更多证据，不猜
3. **3次失败规则** — 3个假设都错就STOP，问用户：
   - A) 继续调查（有新假设）
   - B) 升级人工审查
   - C) 加监控等下次捕获

## Phase 4: 实施

1. 修根因，最小变更
2. 写回归测试：**无fix时失败**（证明确实抓到bug），**有fix时通过**
3. 跑全量测试
4. 修>5个文件 → 提醒用户评估影响范围

## Phase 5: 验证报告

```
DEBUG REPORT
════════════════════════════
症状: [用户观察到什么]
根因: [真正错在哪]
修复: [改了什么，file:line]
证据: [测试输出，复现结果]
回归测试: [file:line]
相关: [同区域历史bug]
状态: DONE | DONE_WITH_CONCERNS | BLOCKED
════════════════════════════
```

## 经验沉淀
Debug中发现非显而易见的模式/陷阱 → `slm remember "问题→根因→方案" --tags debug-pattern`
