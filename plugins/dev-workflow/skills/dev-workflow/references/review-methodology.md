# Review 方法论 v6 — 6角色多视角审查

> v6.0.0 | 借鉴：gstack CEO/Eng/Design/QA/Security/Release 角色分工

---

## 6角色定义

### CEO 🎯（战略视角）
- 这个改动是否与项目目标一致？
- 是否有更简单的替代方案？
- 是否在解决正确的问题？
- ROI：投入的开发时间 vs 收益

### Eng 🔧（工程视角）
- 数据流完整性：happy path + 3个 shadow path（nil/空/上游错误）
- 边界条件：重复操作、中途取消、慢连接、过期状态
- 错误命名：说出具体异常名、触发条件、用户看到什么
- 性能：N+1查询、内存泄漏、不必要的重渲染

### Design 🎨（设计视角）
- API设计是否直觉、一致？
- 接口命名是否自解释？
- 新增接口是否与现有风格统一？
- 用户/开发者体验是否合理？

### QA 🧪（质量视角）
- 测试覆盖率是否充分？
- 边界场景是否有测试？
- 是否会引入回归？
- 错误路径是否有测试？

### Security 🔒（安全视角）
- OWASP Top 10 检查
- 输入验证边界
- 信任边界：LLM输出、外部API是否被信任
- 密钥/敏感信息泄露

### Release 🚀（发布视角）— 条件触发
- 版本号是否正确更新？
- CHANGELOG 是否记录？
- 是否有破坏性变更（breaking change）？
- 迁移路径是否文档化？

## 模式 × 角色

| 角色 | Standard | Full | 触发条件 |
|------|----------|------|---------|
| CEO 🎯 | ✅ 始终 | ✅ 始终 | — |
| Eng 🔧 | ✅ 始终 | ✅ 始终 | — |
| Design 🎨 | ❌ | ✅ | 有API/UI变更时 |
| QA 🧪 | ❌ | ✅ | 有测试变更时 |
| Security 🔒 | ❌ | ✅ | Full模式 |

## 置信度校准

| 分数 | 含义 | 处理 |
|------|------|------|
| 9-10 | 已验证bug | 正常展示 |
| 7-8 | 高置信 | 正常展示 |
| 5-6 | 可能误报 | 附注"需验证" |
| 3-4 | 可疑 | 仅附录 |
| 1-2 | 推测 | 仅P0才报告 |

---

## 发现格式

```
[严重度] (置信度: N/10) [角色] file:line — 描述
```

严重度：P0(紧急) / P1(高) / P2(中) / P3(低)
角色标签：🎯CEO / 🔧Eng / 🎨Design / 🧪QA / 🔒Security / 🚀Release

## 审查产出模板

```markdown
## Code Review Report

- **模式**: Standard/Full
- **审查角色**: CEO + Eng [+ Security]
- **文件数**: N | **发现数**: N
- **P0**: X | **P1**: X | **P2**: X | **P3**: X

### 发现列表

[P0] (9/10) 🔧Eng src/utils.ts:42 — 数据库连接未关闭，长时间运行会泄漏
[P1] (7/10) 🎯CEO src/api.ts — 可以用缓存替代重复查询，更简单
[P2] (5/10) 🎨Design src/types.ts — 接口命名IUser与项目风格User不一致

### 建议

- P0问题需要立即修复（→ 回Step 7）
- P1问题建议修复
- P2/P3可后续处理
```

---

## 修复优先规则

1. 修根因不修症状
2. 最小diff
3. 不顺手重构周边代码
4. 修>5个文件时提醒用户评估影响范围
5. 每个角色独立看代码，不互相影响判断

## 经验对比 ⭐v6

审查时自动对比 `references/lessons/` 中已知经验：
- 检查是否犯了历史项目记录的同类错误
- 发现匹配时标注：`⚠️ 历史经验：<来源项目> 曾遇到相同问题`

---

## 审查后

- 小问题（P2-P3）→ 自动修复
- 大问题（P0-P1）→ 问用户
- 审查产生修改 → 回到 Step 7 重新开发
- 所有发现记录 → Step 12 经验沉淀
